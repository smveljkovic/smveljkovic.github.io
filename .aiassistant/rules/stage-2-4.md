Reviews collection and dynamic review pages

There is an Astro content collection for reviews.

Review Markdown files live in:

```text
src/content/reviews/
```

Existing generated review pages include at least:

```text
src/content/reviews/cosmic-connections.md
src/content/reviews/christian-right-europe.md
```

The dynamic route is:

```text
src/pages/publications/reviews/[slug]/index.astro
```

Expected generated URLs:

```text
publications/reviews/cosmic-connections/
publications/reviews/christian-right-europe/
```

The old manual Cosmic Connections page was removed/moved from `src/pages`, so it
no longer masks the dynamic route.

The dynamic route loads review entries using `getCollection("reviews")`, renders
the Markdown content, and passes `entry.data` into review components.

In components such as `ReviewIntro.astro`, the prop called `review` is already
the frontmatter/data object. Therefore fields are accessed as:

```ts
review.title
review.citationHtml
review.openingVersionNote
review.publishedReview
```

not:

```ts
review.data.title
```

unless the full content entry is deliberately being passed.

Review intro and byline modelling

The review intro needed to vary by version type. The fixed text:

```text
The following text is the Original Submission for
```

was replaced by a frontmatter-driven field:

```yaml
openingVersionNote: "the Original Submission"
```

or similar.

This field is required, not optional, because every review page needs it.

There is also a `ReviewByLine.astro` component. Because the byline needs to
reproduce historical/version-specific wording exactly, the current pragmatic
solution is to store byline display HTML in frontmatter, e.g. `bylineHtml`, and
render it via the component.

This is acknowledged as inelegant but acceptable for the current migration
baseline. Longer term, the byline model may be refactored into more structured
fields.

General principle adopted:

- **structured metadata** powers JSON-LD, sorting, and long-term reuse;
- **display HTML fields** such as `citationHtml`, `bylineHtml`, and `noteHtml`
are acceptable temporary bridges for preserving exact visual/historical
formatting.

Review JSON-LD schema factory

A reusable factory was created:

```text
src/data/schema/reviews/createReviewSchema.ts
```

The dynamic review route now uses:

```ts
const jsonLd = createReviewSchema(review);
```

and passes that to the layout:

```astro
<ReviewLayout ... jsonLd={jsonLd}>
```

This replaced the old pattern of having one hard-coded schema file per review.

Old/manual schema files, such as the earlier Cosmic Connections schema, may
still exist as reference material, but they are no longer the desired pattern.

The conceptual aim of `createReviewSchema()` is to generate a graph with
distinct nodes for:

1. the author/person, identified by ORCID;
2. the reviewed work, preferably identified by DOI;
3. the formally published review, if one exists, preferably identified by DOI;
4. the local review page/version on my website;
5. later, optionally, journal/periodical, volume, and issue nodes.

Important JSON-LD modelling principle:

- the **local website version** and the **published DOI version** should remain
distinct;
- the local version can refer to the published version using `isBasedOn` and/or
`citation`;
- DOI URLs should be used as stable `@id`s where available;
- my author identity should use the ORCID `@id`.

Stage 2.3 is considered substantially complete because:

- both generated review pages emit JSON-LD via `createReviewSchema(review)`;
- no new per-review `.ts` schema file is needed;
- the local page, reviewed work, author, and published review are represented as
distinct graph nodes.

Remaining possible JSON-LD refinements include:

- add `version` back to the local review node;
- distinguish reviewed-work image from published-review image;
- consider separate `WebPage` and `Review` nodes later;
- add journal/volume/issue hierarchy;
- improve handling of edited volumes using `editor` rather than pretending the
editor is the `author`.

For edited volumes such as *The Christian Right in Europe*, the better semantic
model is:

```yaml
reviewedWork:
  editor:
    name: "Gionathan Lo Mascolo"
```

while optionally preserving a display string such as:

```yaml
author: "Gionathan Lo Mascolo (ed.)"
```

for current page titles or visible citations if needed.

Publications page, Stage 2.4

The `/publications/` page is now being converted to generate entries from
collections.

Current implementation loads the reviews collection:

```ts
const reviews = await getCollection("reviews", ({ data }) => !data.draft);
```

and maps entries into display objects using `citationHtml` as an intermediate
display source.

The current strategy is intentionally conservative:

- preserve the old visual design as close to 100% as possible;
- keep using the old classes such as `BibEntry` and `counter_bib`;
- render citation strings using `set:html={item.citationHtml}`;
- group entries by year;
- sort using frontmatter fields such as `publicationList.year`,
`publicationList.sortDate`, and `publicationList.order`.

This has the advantage that citations do not need to be regenerated
programmatically yet. It preserves exact punctuation, italics, DOI wrapping, and
idiosyncratic bibliographic formatting.

A `publicationList` frontmatter object is being used/considered for review
entries:

```yaml
publicationList:
  year: 2025
  sortDate: "2025-05-01"
  order: 40
  noteId: "cosmic-connections-note"
  noteHtml: >-
    The original, unabridged version of this review may be found here:
    <a href="/publications/reviews/cosmic-connections/">html</a> |
    <a href="/publications/reviews/cosmic-connections/file.pdf">pdf</a>.
```

This restores the crucial secondary lines on the publications page linking to
HTML/PDF versions.

Need for lightweight publication-list entries

At present, `/publications/` only shows the two reviews that have full review
content files. That is expected, because it only queries the `reviews`
collection.

But I do **not** want to create full review pages for every publication
immediately.

The agreed strategy is therefore to introduce a lightweight collection for
bibliography/list-only items, perhaps called:

```text
src/content/publication-items/
```

or similar.

Purpose:

- entries appear on `/publications/`;
- they do not necessarily generate their own pages yet;
- they can preserve old bibliography entries via `citationHtml`;
- they can later be enriched with structured metadata for JSON-LD;
- when an item receives a full page later, it can move into a fuller
collection/page model.

This avoids forcing every bibliography item into the `reviews` collection, whose
schema is intended for full review pages.

The `/publications/` page should eventually combine:

1. full review entries from `reviews`;
2. lightweight bibliography-only entries from `publicationItems`.

---

JSON-LD strategy for `/publications/`

I raised the idea of putting JSON-LD for all publication entries on
`/publications/`.

The agreed interpretation is that this is a **sound interim strategy**, not a
replacement for eventually giving each major publication its own page.

The appropriate JSON-LD model for `/publications/` is:

- `/publications/` is a `CollectionPage`;
- its central entity is an `ItemList`;
- the `ItemList` contains `ListItem`s pointing to publication/review/thesis
nodes;
- each publication node should use the most stable identifier available:
  - DOI URL if available;
  - local page URL if it has its own page;
  - otherwise a stable fragment ID under `/publications/`.

Important correction/clarification: the central node is not literally
`collectionList`; the Schema.org model should probably be:

```text
CollectionPage
  mainEntity → ItemList
```

with each listed publication represented through `ListItem`.

This can be implemented later with a separate factory such as:

```text
src/data/schema/publications/createPublicationsSchema.ts
```

Conceptual target graph for `/publications/`:

1. `WebSite` node;
2. `Person` node using ORCID;
3. `CollectionPage` node for `/publications/`;
4. `ItemList` node for the publication list;
5. publication/review/thesis nodes for visible entries.

This would increase the site’s publications signal while remaining honest: the
publications page visibly lists the works, and the JSON-LD describes that
visible list.

The long-term strongest model remains:

- `/publications/` has a well-structured `CollectionPage` + `ItemList`;
- each major item eventually has its own focused page and JSON-LD;
- DOI-bearing objects use DOI URLs consistently as `@id`s across all pages.

Current strategic priorities

Immediate next steps for Stage 2.4:

1. Finish making `/publications/` visually complete.
2. Add a lightweight `publicationItems` collection for bibliography-only
entries.
3. Combine `reviews` and `publicationItems` on `/publications/`.

`CollectionPage` + `ItemList` model.
8. Later enrich lightweight entries with structured metadata sufficient to
generate publication nodes.
9. Eventually give each major publication its own page and migrate its metadata
into a fuller model.

Guiding principle:

Preserve visual fidelity first, while steadily moving the site toward
structured, reusable content and robust JSON-LD.

Stage 2.3 can be treated as essentially done; Stage 2.4 is now the active focus.
